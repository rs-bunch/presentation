Коммуникация внутри команды

Процесс разработки состоял из ежедневных митингов, организация и процесс которых базировался на методологии Agile/Scrum.

В начале каждого спринта, с помощью голосовых вызовов в Discord, производился митинг, на котором обсуждались:

  • Цели спринта
  • Расстановке приоритетов выполнения
  • Готовности участников команды к выполнению той или иной части спринта.

    • В основном каждый выбирал себе часть работы, учитывая свои обязанности в предыдущем спринте и основываясь на своей возможности довести дело до конца в срок.

Реальный пример распределения задач:

  • Допустим участник команды работал над определенным участком проекта в прошлом спринте
  • Основываясь на этом, он выбирал себе задания из нового спринта по развитию или улучшению данного участка проекта.
  • А это значит, что участник выполнял понятную ему работу и видел четкие цели для ее завершения вовремя

Команда проводила ежедневные короткие митинги, на которых обсуждались:

  • Текущее положении разработки каждого участника
  • Возникшие проблемы
  • Согласование по изменению частей проекта, имеющих признаки общего использования (Api, Router, Store).

По результатам обсуждения различных вопросов иногда создавалась мини документация в соответствующих папках дискорд.

Основным рабочим каналом во время спринтов становился канал “code discussion”. В этом канале мы в рабочем режиме могли обсудить ту или иную проблему при написании кода, поделится примером написанного кода и спросить совета по реализации, не дожидаясь при этом митинга.

Не менее значимым являлся канал “git”, интегрированный с репозиторием.

В нём команда автоматически информировалась о событиях в репозитории, таких как открытие, закрытие или мердж PullRequest. Этот канал помогал максимально быстро информировать членов команды о необходимости провести код ревью или о мердже важных изменений.

Обзор технологического стека

Командой было решено использовать, для разработки SPA-приложения, технологию Веб-компонентов. Основной причиной послужило желание участников команде поглубже разобраться в данной тематике, а также наличие у некоторых участников практических навыков работы с ними.

  • Webpack - сборщик модулей JavaScript с открытым исходным кодом
  • Typescript - типизированная надстройка над JS
  • В качестве роутера был разработан компонент, отслеживающий загрузку страницы, а также перемещение по истории и состояние адресной строки
  • Для управления состоянием различных частей проекта мы решили использовать Redux-Toolkit.
  • В качестве взаимодействия с API CommerceTools было принято решение использовать SDK CommerceTools
  • bootstrap - свободный набор инструментов для создания сайтов и веб-приложений
  • toastify - легковесная библиотека уведомлений на JS.
  • husky - библиотека для создания git хуков
  • jest - библиотека для модульного тестирования
  • eslint - инструмент статического анализа кода для выявления проблемных шаблонов, найденных в коде JavaScript
